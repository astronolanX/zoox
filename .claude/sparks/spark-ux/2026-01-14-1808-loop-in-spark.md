## Insights

**The 70% emergent questions the premise.** Thread's inversion pattern persists even in a new thread: we asked "where do loops live in sparks?" and the swarm responded "what if they don't exist there at all?"

**Current spark architecture is linear.** Each spark runs once:
1. Topic → 5D combinatorics → tiers → synthesis → output
2. No iteration, no refinement, no feedback

**Where loops *could* live:**

1. **Between tiers** — Near insight feeds into Mid prompt, Mid feeds Far
2. **In memory retrieval** — Iteratively surface more relevant past sparks
3. **In synthesis** — Convergent/divergent/emergent could feed back for refinement
4. **Across sessions** — Spark threads as loops over time (this thread is proof)

**The "nested stanzas" insight**[^1] is evocative: loops as recursive structure within a poem. Each spark could contain smaller sparks, each tier could iterate before synthesis.

**The "don't exist" frame suggests:** Maybe sparks are intentionally non-looping. The value is in the *single pass* — forcing the swarm to commit without refinement. Loops would add compute but might not add insight.

**Practical options:**

1. **`--deep` mode with iterations** — Run tiers multiple times, synthesize the syntheses
2. **Thread-level loops** — `/spark` on the same topic automatically builds on previous spark
3. **Memory-as-loop** — Past sparks influence current spark (already exists via SQLite)
4. **No loops** — Accept that sparks are one-shot provocations, not refined answers
